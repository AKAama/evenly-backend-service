## 多人协作分账应用 · 核心开发文档 v1
---

## 1. 产品背景与目标
### 1.1 使用场景
用户在多人出行、聚餐、合租等场景中，创建一个**共享账本**，账本内所有成员：

+ 都可以新增支出记录
+ 都可以实时看到所有已确认支出
+ 系统自动计算最终「谁该转给谁多少钱」

典型场景示例：

> Alex、Sylvia、Stella、Tristen 一起去青岛旅行  
Alex 先垫付了午饭 300 元  
系统自动计算：其余三人各转给 Alex 75 元
>

---

### 1.2 核心目标
+ 多人实时共享同一本账
+ 明确每一笔钱的来源与归属
+ 尽量避免现实中的分账争议
+ 结算结果清晰、可解释、可追溯

---

## 2. 系统整体设计原则
### 2.1 设计哲学
> **人可以被管理，钱不可以被篡改**
>

+ 成员可以被邀请、移除
+ 支出一旦确认，即视为财务事实，不可更改

---

### 2.2 数据驱动而非状态驱动
+ **不保存结算结果**
+ 所有结算均由「已确认支出」实时计算得出
+ 数据变化 → 结算自动变化

---

## 3. 模块划分（功能层）
### 3.1 登录与用户系统（基础设施）
+ 用户通过账号体系登录（如邮箱 / 第三方）
+ 每个用户拥有唯一 `userId`
+ 用户信息包括：
    - 显示名
    - 邮箱
    - 头像（可选）

> 用户系统用于**身份绑定**，不参与具体分账规则
>

---

### 3.2 账本（Ledger）
#### 账本属性
+ 唯一 ID
+ 名称（如「青岛之旅」）
+ Owner（创建者）
+ 成员列表
+ 创建时间

---

### 3.3 成员与权限模型（A）
#### 角色划分（仅两种）
##### Owner（账本创建者）
+ 管理账本与成员
+ 不拥有修改他人支出的特权

##### Member（普通成员）
+ 与 Owner 在“钱”的规则上完全平等

---

#### 权限规则
| 行为 | Owner | Member |
| --- | --- | --- |
| 查看所有支出 | ✅ | ✅ |
| 新增支出 | ✅ | ✅ |
| 修改 / 删除自己的支出 | ✅ | ✅ |
| 修改 / 删除他人的支出 | ❌ | ❌ |
| 添加 / 移除成员 | ✅ | ❌ |
| 解散账本 | ✅ | ❌ |
| 退出账本 | ❌ | ✅ |


---

### 3.4 成员邀请机制（简化版）
+ 添加成员时，直接基于 **用户身份（如邮箱 / uid）**
+ 被添加成员 **无需确认**
+ 添加后立即可见账本与历史支出

> 第一版假设用户均为熟人，刻意简化流程
>

---

## 4. 支出模型与确认机制（B）
### 4.1 支出（Expense）
每一条支出至少包含：

+ 金额（以“分”为单位）
+ 付款人（payer）
+ 参与人列表
+ 创建者
+ 状态（见下）

---

### 4.2 支出状态机
```latex
Pending   待确认
Confirmed 已确认（参与结算）
Rejected  被否决（不参与结算）
```

---

### 4.3 状态流转规则
+ 新建支出 → Pending
+ 所有成员确认 → Confirmed
+ 任意成员否决 → Rejected

---

### 4.4 确认规则
+ 所有成员（包括创建者）都必须确认
+ 每个成员对每条支出只能操作一次（确认或否决）
+ 已 Confirmed 的支出：
    - 不可修改
    - 不可删除

---

### 4.5 否决机制说明
+ 否决 ≠ 修改
+ 否决后该支出不参与结算
+ 原创建者可：
    - 删除该记录
    - 或新建一条修正记录

---

## 5. 结算模型（C）
### 5.1 结算原则
+ 仅统计 **Confirmed** 状态的支出
+ 结算结果为「转账指令集合」
+ 结算结果不持久化

---

### 5.2 核心计算逻辑
#### Step 1：计算应承担金额
+ 每条支出按参与人均分
+ 累加得到每人应承担金额

---

#### Step 2：计算已垫付金额
+ 累加每人作为付款人的支出金额

---

#### Step 3：计算净额（Net Balance）
```latex
net = 已垫付 - 应承担
```

+ net > 0：应收钱
+ net < 0：欠钱

---

### 5.3 转账指令生成（贪心算法）
+ 将用户分为：
    - 债权人（net > 0）
    - 债务人（net < 0）
+ 反复执行：
    - 债务人向债权人转账 `min(欠款, 应收)`
    - 清零的一方移出队列
+ 直到全部结清

---

### 5.4 输出形式
```latex
A → B：X 元
```

示例：

```latex
Sylvia → Alex 75
Stella → Alex 75
Tristen → Alex 75
```

---

## 6. 成员退出与账本解散
### 6.1 成员退出
+ 不可再新增支出
+ 历史支出仍参与结算

---

### 6.2 Owner 解散账本
+ 账本整体关闭
+ 历史数据保留（只读）
